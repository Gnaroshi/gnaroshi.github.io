# English And Korean Localization

## URL Policy

- English is the default locale and keeps the existing unprefixed URLs, such as `/blog/` and `/papers/`.
- Korean uses the `ko` locale and the `/ko/` prefix, such as `/ko/blog/` and `/ko/papers/`.
- Do not add `/en/` or `/kr/` routes.
- Never redirect solely from `navigator.language`. The language switch records an explicit choice but navigation remains user initiated.

Astro's i18n configuration is in `astro.config.mjs`. Locale types, dictionaries, routing helpers, date formatting, and tag labels live under `src/i18n/`.

## Page Architecture

Root and `/ko/` route files are thin wrappers. Shared page implementations live in `src/views/` and receive a typed `locale` prop. Business logic, collection queries, and dashboard calculations must remain shared.

`BaseLayout.astro` owns the document language, localized shell, canonical URL, `hreflang` links, Open Graph locale, header, and footer. React islands receive both `locale` and a typed subset of `IslandMessages`; do not import an English-only label table into an island.

## Static Data

Locale-aware profile, research, project, skill, and current-focus data live in `src/data/locales/`. English may reference the canonical source data. Korean copy must preserve the same facts and omit unknown fields rather than inventing values.

## Localized Content

Every content entry includes:

```yaml
locale: "en"
translationKey: "stable-concept-id"
translationStatus: "complete"
```

Allowed translation statuses are `complete`, `partial`, and `source-only`. Papers also require a language-neutral `paperId` so notes about the same paper can be paired even when their slugs differ.

Translated blog files use matching paths:

```text
src/content/blog/en/paper-reading-method.mdx
src/content/blog/ko/paper-reading-method.mdx
```

The locale directory is removed from the public slug. Pair entries with `translationKey`, not filename assumptions. UI is always translated, but technical content may remain source-only. If no counterpart exists, the language switch links to the localized collection index and labels the translation as unavailable. Never render an unmarked English article at a Korean URL.

## Writing Korean Copy

- Translate meaning and tone, not English sentence structure.
- Keep paper titles, model names, dataset names, code, formulas, GitHub, Astro, React, and MDX unchanged unless an official Korean name exists.
- Keep canonical tag IDs language-neutral and localize display labels in `src/i18n/tags.ts`.
- Use `formatLocalizedDate` and `formatLocalizedNumber`; do not hand-format dates or counts.
- Follow `docs/i18n-terminology.md` for recurring product language.

## Verification

Run the build before generated-output checks:

```bash
npm run check
npm run check:i18n
npm run check:hardcoded-ui
npm run build
npm run check:translation-links
npm run test:e2e
```

`check:i18n` enforces exact key parity, non-empty values, placeholder rejection, and accidental English-copy detection. `check:hardcoded-ui` audits localized islands and Korean output. `check:translation-links` validates built route pairs, canonical URLs, `hreflang`, RSS, and sitemap output.
